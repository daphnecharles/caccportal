PC's

  • Hardware
    • Keyboard/Mouse
      1. Check cables.
      2. Unplug and plug back in.
      3. Restart machine.
      4. Plug into adjcent computer.
        1. If keyboard works, then submit a ticket in HDRT (Computer Repair Queue) for that computer.
        2. If keyboard does not work, then submit a ticket in HDRT (Computer Repair Queue) for keyboard replacement.
          1. Follow up with Computer Repair and replace with a backup.
    • Monitor
      1. Check cables.
      2. Unplug and plug both ends of DVI cable back in.
      3. Restart machine
      4. Plug into adjacent machine
        1. If monitor works, then submit a ticket in HDRT (Computer Repair Queue) for that computer.
        2. If monitor does not work, then submit a ticket in HDRT (Computer Repair Queue) for that computer.
      5. Try adjusting display settings if projector refuses to display an image.
    • Power
      1. Check to make sure power cord is plugged properly from both ends.
      2. Check surge protectors
      3. Plug the power cord into another outlet. If a wall outlet is deemed faulty contact facilities.
      4. Check power cord with another machine.
        1. If power cord does not work on another computer, then replace power cord.
        2. If power cord does work then submit, then submit a ticket in HDRT (Computer Repair Queue) for that compuer.
          1. Follow up with Computer Repair
    • USB Port
      1. See if device works on another USB port and on another mahcine.
        1. If device works then USB port is bad and a Computer Repair ticket should be submitted in HDRT.
        2. If device does not work, then the device is bad.
    • Network
      1. Check to see if there is a link light.
        1. Yes, then there could be something wrong with the network. Try pinging the machine from another machine to see if there is a reply.
        2. No.
          1. Swap network cables with another machine and check symptoms on each machine.
          2. Check BIOS time.
          3. Check to make sure the server rack is up and running.
  • Software
    • Rebuild
      • This information is also covered in the NetOps Wiki, the link can be found on the Supervisor Page.
      • Joe can now initiate a rebuild of any PC remotely.
      • To start this process, simply enter a ticket in HDRT with the machine name and reason for rebuild.
      • For HP's submit a ticket in HDRT in NetOps PC Queue.
    • Key Server
      1. There are only a limited number of licenses available per software.
      2. If user gets a message that there are not enough licenses, they must wait until one is available.
      3. Software licensing is controlled by the Key Server.
      4. There is a central Key Server in Hill and local one on each campus for redundancy.
      5. If the Key Server cannot be contacted then a dialog box "Searching for Shadows" will appear when launching applications.
      6. If the PCs are "Searching for Shadows" then wait until the request to the central key server times out and a key is granted from the local key server.
      7. Confirm that the Key Server is down by testing multiple PCs.
      8. Place a ticket in HDRT (EMERGENCY Queue) if the Key Server is not responding.
    • Authentication
      • This information is also covered in the NetOps Wiki, the link can be found in the NetOps section.
      1. PCs use LDAP authentication.
      2. The LDAP servers are in Hill Center.
      3. If network is experiencing problems, then the PCs will not be able to login.
      4. If more than one PC is not able to login, then submit a ticket in HDRT. Depending on the severity of the problem, you may want to submit a ticket in either the PC queue or Emergency queue.
      5. If only one machine cannot log in:
        1. The system time in the BIOS may be wrong. Verify and correct as necessary.
        2. The machine's network cable may be damaged (see PC -> Hardware -> Network).
        3. The machine may need to be rebuilt.
  • Other
    • Password Protecting Bios
      1. All PC BIOSs must be password protected.
      2. If password is removed, you must remember to reapply the password.
      3. All new machines must be passworded.
      4. Machines coming back from Computer Repair need to have passwords set.
    • Boot Sequence
      • The boot sequence under normal operating conditions:
        1. First boot device is "Integrated NIC".
        2. If this option is not available, follow below:
          1. Go to BIOS.
          2. Go to "Integrated Devices".
          3. Go to "Network Interface Controller".
          4. Change value to "On w/ PXE".
        3. All other devices should be disabled

Printing

  • Common Fixes
    • Restarting the printer can solve 50% of problems.
    • Checking the physical connections is always a good idea
    • Patrons accidently press the "Pause/Go" button frequently.
    • Always check pickup rollers and fuser whenever the printers jam.
    • A good cleaning will keep the repair supervisor happy.
  • Hardware
    • Kyocera 9530's
      • Rebooting the printer will often times fix a problem, so make sure you try that as the first step.
      • Try moving or deleting pending jobs for a printer from the queue. Occasionally a rogue print job will cause error messages that may appear to be a hardware problem.
      • Call ICPNJ to request service for any problems that can not be fixed by lab personel
    • Kyocera 9520's
      • Rebooting the printer will often times fix a problem, so make sure you try that as the first step.
      • Try moving or deleting pending jobs for a printer from the queue. Occasionally a rogue print job will cause error messages that may appear to be a hardware problem.
      • Call ICPNJ to request service for any problems that can not be fixed by lab personel
    • Kyocera 5030N Color Printer
      • Rebooting the printer will often times fix a problem, so make sure you try that as the first step.
      • Try moving or deleting pending jobs for a printer from the queue. Occasionally a rogue print job will cause error messages that may appear to be a hardware problem.
      • Call ICPNJ to request service for any problems that can not be fixed by lab personel
      • The duplexer on this machine is located beneath the paper tray and slides out like the paper tray
  • Network
    • Physical Check
      1. Restart printer.
      2. Check physical connections for printer.
      3. Test RJ45 cable.
      4. Test with different RJ45 cable. Restart the printer.
      5. Test original RJ45 cable on another printer.
      6. Verify that the ethernet port is working.
      7. Depending on diagnosis:
        • Cable might need to be replaced.
        • Enthernet port might be bad. Consult managers for guidance as to how to proceed. (Contact TD/submit ticket in HDRT)
        • Ethernet card in printer might be bad. Submit ticket in HDRT (Computer Repair Queue).
      8. Any changes will require restarting the printer.
    • Software Check
      1. Print out configuration page and verify that the printer is receiving DHCP information.
      2. Ping the printer's IP address and verify network connectivity
      3. Depending on diagnosis a ticket in HDRT may need to be submitted. (This is for HP's, call ICPNJ for Kyocera's)
      4. If all printers are not functioning, place an EMERGENCY ticket in HDRT.
  • Print Release Stations/Pharos Remote
    • General Troubleshooting Tips
      1. If the release station is not swiping any patrons cards, try switiching the port that the USB cable is plugged into.
      2. You can also try cleaning the card reader with the cleaning cards that are located in the TD room.
      3. Make sure the user is logged in under their own username, otherwise they will not be able to release their jobs when swiping.
      4. If you get a complaint from a user, the first thing you should do is test it out yourself. Try logging into a machine and try to print something. Then try to release it. Remember that Guest users and some others will not have the ability to use the Release Station. They might go up to the station trying to find their job, but it has already printed out on a printer.
      5. If there is a problem with the Release Station, you can always release/move the jobs via Pharos Remote. Don't be afraid to reboot the Release Station. Just hit the power button on the front (just once, don't hold it down) and it will shut down Windows. Then hit it again to power it back up. If it doesn't load the release station software, then there might be a problem and an emergency ticket should be created. It will give an error describing why it can't load. Please note this in the ticket.
      6. Try printing to a different queue and see if you can release the job from the associated Release Station.
      7. Try waiting a few minutes. If there is a network glitch, it might take some time to reconnect. If you swipe a card and it says "communication error" or "Trying to contact DBServer," then give it some time. Call a lab on another campus to see if they are affected as well. This will allow you to determine if it's a campus-wide problem or if it's local to your lab. If it persists for more than 2-3 minutes, submit an emergency ticket.
      8. If the print release station hangs on "Reconfiguring...please wait," then reboot it. It won't take longer than 2-3 seconds to reconfigure. If it hangs, that means there's a problem and it needs to be rebooted.
Networking
  • PC's
    • This information is also covered in the NetOps Wiki.
    • Normal operation conditions:
      1. PC first talks to "Baby Eden" (local server at each site).
      2. If username is valid and allowed to login, then PC talks to the Kerberos Server in the Hill Center.
      3. If password is authenticated without problems, then user is granted login.
    • No connectivity beyond College Avenue Campus:
      1. PC will talk to "Baby Eden" (local server at each site).
      2. If udername is valid and allowed to login, then PC talks to Kerberos Server in the Hill Center.
      3. If Kerberos Server cannot be conacted, request will timeout and the user will be granted login without password authentication.
    • No connectivty with local server (ie, server rack, CCF-cac).
      1. PC will talk to "Baby Eden" (local server at each site).
      2. If "Baby Eden" cannot be contacted, request will timeout and the user will be denied login.
  • Mac's
    • This information is also covered in the NetOps Wiki.
    • Normal operating conditions:
      1. Mac will talk with LDAP server in the Hill Center.
      2. If username and password are authenticated without problems then the user is granted login.
    • No connectivity beyond College Avenue Campus:
      1. Mac will attempt to talk with the LDAP server in the Hill Center.
      2. If the LDAP server cannot be contacted, the request will timeout and the user will be denied login.
  • IMPORTANT NOTE
    • Machines will take precedence of RCI authentication over EDEN authentication.
    • Instruct patrons with both account types to use RCI authentication.

      • Emergency Tickets
        • Network outage, no printing available, no login available, Key Server issues, and other server problems are valid emergency situations.
        • All NetOps fulltime staff get paged when placing an emergency ticket.
        • Placing emergency tickets:
          1. First line of ticket should be callback phone number.
          2. Next line should be a VERY, VERY brief description of problem (only a few words, that's it).
          3. Lastly, NO signature.

    • Hardware Support
      1. Server Cabinet
        • Any problems with any of the equipment in the server cabinet must be reported to NetOps with a ticket.
        • Do not open/service the server cabinet unless instructed to do so by NetOps.
      2. Scanner
        • On the PC, if Windows tries to reinstall the drivers for the scanner, change the USB port until Windows stops asking for drivers.
        • Any other problems with the scanner, submit a ticket for the respective queue.
      3. Mice
        • All Microsoft branded Mice are managed by NetOps.
        • Send a ticket for replacement mice if on-hand supply of replacements is low.

        • NetOps is responsible for all servers and switches as well as cabling between the switches and patch panels and the machines themselves.
